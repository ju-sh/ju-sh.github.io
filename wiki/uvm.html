Universal Verification Methodology (UVM)

import uvm_pkg::*;
`include "uvm_macros.svh"

New

  • Agent
    • Has everything related to a particular interface
      • Sequencer: Generates stimulus
      • Driver: Drives signals
      • Monitor: Observes activity on the interface and produce transaction objects
    • Driver and monitor are connected to the interface

                                                                 +---->--- coverage
            stimulus                                             |
Sequencer -----------> Driver -----> DUT ------> Monitor --------+
                                                                 |
                                                                 +---->--- scoreboard

From Practical UVM:

think of an agent as a class object that encapsulates everything to do with a specific interface.

  • Env class
    • Contains all sub-components of a testbench
    • Agents are instantiated within an env
    • Also has: coverage monitor, scoreboard
  • config database: essentially set of name-value pairs
  • virtual interface: like a pointer to an actual interface
  • one agent for each DUT interface??
  • Pin wiggles: Low-level signal activity on DUT pins (individual bits toggling) as opposed to high-level transactions. (src: chatgpt)

Conventions

  • vi suffix: virtual interface
  • if suffix: interface
  • h suffix: handle (/instances?)

UVM Basics (verificationacademy)

6: Sequences and tests

uvmsequencer

Inline constraints

assert( tx.randomize() );
tx.cmd = WRITE; tx.addr = a; tx.data = d;

is effectively same as:

assert(
  tx.randomize() with {
    cmd == WRITE; addr == a; data == d;
  }
);

Example

// Extends `uvm_sequence`
// Parameterized by `my_transaction`, which is a transaction type
class read_modify_write extends uvm_sequence #(my_transaction);
  // Register with factory
  `uvm_object_utils(read_modify_write)

  // Constructor
  function new (string name = "");
    super.new(name);
  endfunction: new

  task body;
    // Declare a value of my_transaction class
    my_transaction tx;

    int a;
    int d;

    // Make a value of my_transaction class via factory
    tx = my_transaction::type_id::create("tx");

    // Start setting up the tx
    // `start_item` and `finish_item` helps to do a form of handshaking with driver
    start_item(tx);

    // Randomize the transaction value
    assert( tx.randomize() );

    // READ is an enum valu
    tx.cmd = READ;

    // tx is ready to be sent off to the driver
    finish_item(tx);

    // Obtain values from the tx
    a = tx.addr;
    d = tx.data;
    ++d;

    // A new tx to write updated value
    tx = my_transaction::type_id::create("tx");
    start_item(tx);
    assert(
      tx.randomize() with {
        // WRITE is another enum value
        cmd == WRITE; addr == a; data == d;
      }
    );
    finish_item(tx);

  endtask: body
endclass: read_modify_write

The enum could be like:

typedef enum {
  READ,
  WRITE
} rw_t;

–

Sequences may be nested. For example, we can have another sequence that repeats another sequence n times:

class seq_of_seqs extends uvm_sequence #(my_transaction);
  `uvm_object_utils(seq_of_seqs)

  // n controls how many times the inner sequence is repeated
  // Variables like `n` are nicknamed a 'knobs'
  rand int n;

  constraint how_many { n inside {[2:4]}; }

  task body;
    repeat(n)
    begin
      read_modify_write seq;
      seq = read_modify_write::type_id::create("seq");

      // `m_sequencer` is a built-in name.
      // Means sequencer instance on which the sequence is running
      seq.start( m_sequencer, this);
    end
  endtask: body
endclass: seq_of_seqs

–

package my_sequences;

uvm_test

class test1 extends uvm_test;
  // Register test1 as a component
  `uvm_component_utils(test1)

  // Instantiate an env
  my_env my_env_h;

  task run_phase(uvm_phase phase);
    read_modify_write seq;
    seq = read_modify_write::type_id::create("seq");

    // Raise object till test is ready
    // Objections are usually raised in tests, rather than drivers or sequencers
    phase.raise_objection(this);

    // We can set up stimulus for the test here, if needed

    // The env has an agent, which has a sequencer 
    // This is the currently relevant sequencer  
    // Assumptions here:
    //  - my_env has an agent named my_agent_h
    //  - my_agent_h, in turn has a sequencer named my_sequencer_h
    //
    // This was a top-level sequence, so starting it directly from the test.
    // Hence, there is no parent argument
    seq.start(my_env_h.my_agent_h.my_sequencer_h);

    // seq should now be complete.
    // Test is now ready to fire. Drop objection
    phase.drop_objection(this);
  endtask
endclass 

class test2 extends uvm_test;
  `uvm_component_utils(test2)
  my_env my_env_h;

  task run_phase(uvm_phase phase);
    seq_of_commands seq;
    seq = seq_of_commands::type_id::create("seq");

    // Change setting to allow overriding of constraint
    seq.how_many.constraint_mode(0);

    // Randomize with constraint n overriden
    assert( seq.randomize() with {seq.n > 10 && seq.n < 20;});


    phase.raise_objection(this);
    seq.start( my_env_h.my_agent_h.my_sequencer_h );
    phase.drop_objection(this);
  endtask
endclass

7: Monitors and subscribers

  • Passive components
  • Performs coverage collection and checking
  • Monitor: make transactions out of dut pin wiggles and send it to the verification env
    • Uses its analysis port to send info to its subscribers
    • Subscriber would have an anlysis export
    • Monitor can have any number of subscribers
class my_monitor extends uvm_monitor;
  `uvm_component_utils(my_monitor)

  // Declare an analysis port, parametrized by the type of transactions
  uvm_analysis_port #(my_transaction) aport;

  // A 'pointer' to the DUT interface
  virtual dut_if dut_vi;

  function new ...

  function void build_phase(uvm_phase phase);
    // Can use constructor directly,
    // because we don't need the 'flexibility' of factory for analysis ports
    aport = new("aport", this);
  endfunction

  task run_phase(uvm_phase phase);
    forever
    begin
      my_transaction tx;

      // Wait till a positive clock tick
      @(posedge dut_vi.clock);

      tx = my_transaction::type_id::create("tx");

      // Extract values from the DUT pins 
      tx.cmd = dut_vi.cmd;
      tx.addr = dut_vi.addr;
      tx.data = dut_vi.data;

      // Write tx to the analysis port
      aport.write(tx);
    end
  endtask

// Monitor is inside the agent
class my_agent extends uvm_agent;

  uvm_analysis_port #(my_transaction) aport;

  function void build_phase(uvm_phase phase);
    aport = new("aport", this);
    my_sequencer_h = my_sequencer::type_id::create ...
    my_driver_h = my_driver ::type_id::create ...
    my_monitor_h = my_monitor ::type_id::create ...
  endfunction: build_phase

  function void connect_phase(uvm_phase phase);
    // Connect aport of monitor to the aport of its agent
    my_monitor_h.aport.connect(aport);
  endfunction: connect_phase
endclass: my_agent

// Agent will pass the tx to its env
class my_env extends uvm_env;
  my_agent my_agent_h;

  // Declare a subscriber
  // NQ: Perhaps we can declare many subscribers here?
  my_subscriber my_subscriber_h;

  function void build_phase(uvm_phase phase);
    my_agent_h = my_agent ::type_id::create ...
    my_subscriber_h = my_subscriber::type_id::create ...
  endfunction: build_phase

  function void connect_phase(uvm_phase phase);
    // Connect aport of agent to analysis export of the subscriber
    my_agent_h.aport.connect(my_subscriber_h.analysis_export);

    // NQ: Subscriber will provide impl of the write method called in the monitor?
  endfunction: connect_phase
endclass: my_env


class my_subscriber extends uvm_subscriber #(my_transaction);
  // uvm_subscriber already has an analysis export, so we can inherit that
  `uvm_component_utils(my_subscriber)

  bit cmd;
  int addr;
  int data;

  // Covergroup: Defines what we are looking for.
  // Based on the verification plan.
  covergroup cover_bus;
    coverpoint cmd;
    coverpoint addr { bins a[16] = {[0:255]}; }
    coverpoint data { bins d[16] = {[0:255]}; }
  endgroup: cover_bus

  function void write(my_transaction t);
    cmd = t.cmd;
    addr = t.addr;
    data = t.data;

    // Record the information in the cover database??
    cover_bus.sample();
  endfunction: write
endclass: my_subscriber

8: Reporting

  • UVM has built-in reporting
  • Can use macros to make messages
  • `uvm_info("originator/id", "Message", UVM_VERBOSITY)
  • Eg: `uvm_info("mg", "Transaction received", UVM_NONE)
    • Will get # UVM_INFO filename.sv(110) @ 100: uvm_test_top.m_env.m_driver [mg] Transaction received
    • 110: line number
    • 100: simulation time

Severity

  • `uvm_info ("id", "message", verbosity)
  • `uvm_warning("id", "message")
  • `uvm_error ("id", "message")
  • `uvm_fatal ("id", "message")

Verbosity levels:

  • UVM_NONE: Always appear
  • UVM_LOW
  • UVM_MEDIUM
  • UVM_HIGH
  • UVM_FULL: Usually won't appear

Setting actions:

  • Via report handler
  • uvm_top.set_report_severity_action_hier(UVM_INFO, UVM_NO_ACTION);
    • For the entire hierarchy, take no action for uvminfo messages
    • ie, suppress all uvminfo messages
  • uvm_top.set_report_id_action_hier(“mg", UVM_NO_ACTION);
    • For the entire hierarchy, take no action for messages with id "mg"
    • ie, suppress all messages with id "mg"
  • NB: The _hier methods must be called after build phase
    • Calling them just before start of simulation would be good

Actions:

Name Action
UVM_NO_ACTION Suppress the messages
UVM_DISPLAY Send report to standard output (default)
UVM_LOG Send report to a file
UVM_COUNT Stop simulation when max count is reached (error)
UVM_EXIT Finish simulation immediately (fatal)
UVM_STOP Call $stop

UVM Advanced

UVM macros

https://verificationacademy.com/verification-methodology-reference/uvm/docs_1.1c/html/files/macros/uvm_object_defines-svh.html

  • `uvmobjectutils:
    • Register a class with UVM factory
    • Support randomization for sub-classes of uvmsequenceitem
    • Eg: `uvm_object_utils(my_item (Note: no semicolon at end)
  • `uvm_object_utils_begin

    • Register fields of a class with UVM to automatically generate boiler plate like definitions for convert2string, copy, compare, print, etc

    • Example:

      class my_item extends uvm_sequence_item;
  rand int ip1;
  rand int ip2;

  `uvm_object_utils_begin(my_item)
    `uvm_field_int(ip1, UVM_ALL_ON)  // for integers
    `uvm_field_int(ip2, UVM_ALL_ON)  // for integers
  `uvm_object_utils_end

  function new(string name="my_item");
    super.new(name);
  endfunction
endclass

    • UVMALLON: means this field is used for

      • Copying (docopy)
      • Printing (doprint)
      • Comparison (docompare)
      • Recording and randomization (dopack/dounpack if needed)
  • `uvm_object_param_utils: register parameterized classes with UVM

    • Example:

      class adder_transaction #(int WIDTH = 4) extends uvm_sequence_item;
  `uvm_object_param_utils(adder_transaction#(WIDTH))

  rand logic [WIDTH-1:0] a, b;
  rand logic cin;
  logic [WIDTH-1:0] s;
  logic cout;

  function new(string name = "adder_transaction");
      super.new(name);
  endfunction
endclass
  • `uvm_component_utils: like `uvm_object_utils but for components
    • For entities like driver, monitor, agent, env, tests
  • `uvm_do:

    • simplifies starting and finishing a sequence item in a sequence.

    • Kind of wraps calls to startitem, randomize, finishitem into one line.

      adder_transaction tx;
tx = adder_transaction::type_id::create("tx");

`uvm_do(tx);

/*
// Tell sequencer we're about to use tx
start_item(tx);

// Randomize the transaction
assert(tx.randomize());

// Send it to driver
finish_item(tx);
*/

Component vs object

  • Object: Data
    • Just holds the values
    • Eg: uvmsequenceitem
    • No presence in run/build/connect phases
    • No code execution involved
    • Creation: new()
  • Component
    • Creation: factory type_id::create()
    • Eg: driver, monitor, sequencer
    • Involved in run/build/connect phases
    • Can execute code
    • There is a hierarchy involved. A component have other components.

uvm_analysis_imp

  • UVM analysis implementation
  • Component-side implementation of an analysis interface/port.
  • Receiver side of an analysis port
  • Associated with owner and type of transactions
    • Owner is the component instance that instantiates it

TLM

  • Transaction Level Modeling
  • Communication between components is via TLM, via method calls
  • Initiator and target components
  • TLM port: Specifies the API (kind of like type)
  • TLM export: Provides the implementation
  • Connections are between ports and exports, not directly to the components
    • Initiator's port is connected to target's export
  • Ports and exports are parameterized by the type of transactions used for the communication
class my_env extends uvm_env;
  virtual function void connect();
    initiator.pport.connect(target.pxport);
  endfunction

  class initiator extends uvm_component;
    uvm_put_port #(tr) pport;

    virtual task run();
      pport.put(tr1);
    endtask
  endclass

  class target extends uvm_component;
    uvm_put_imp #(tr,target) pxport;

    virtual task put(tr);
    ...
    endtask
  endclass

  class target2 extends target;
    uvm_put_imp #(tr,target) pxport;

    virtual task put(tr);
      ...
    endtask
  endclass
endclass

New terms

  • SV scheduling regions
  • Clocking blocks provide implicit skew control
  • Explicit skewing:
    • default input #0 output #1ns;: Sample input right away, but drive output 1ns later
    • default input #1ns output #1ns;: Sample input and drive output after 1ns
  • Interface arguments: For signals that come from outside, possibly shared with other components. Eg: clk, rst

Phases

  • Phases (simulation): build, run, cleanup, etc
    • https://vlsiverify.com/uvm/uvm-phases/

—

Kind of specifies the order of tasks

  • Defined as callbacks
  • 9 primary phases
  • function: no simulation time consumed
  • task: can consume simulation time
  • https://www.chipverify.com/uvm/uvm-phases
    • https://vlsiverify.com/uvm/uvm-phases/
  • runphase is the only task, all other phases are functions. ie, run phase is the only simulation time-consuming phase
  • build: construct testbench, create and initialize uvm components
    • extern virtual function void build_phase (uvm_phase phase);
  • connect: connect components in testbench
    • extern virtual function void connect_phase (uvm_phase phase);
  • end of elaboration: final configuration
  • start of simulation: Prepare for initiating simulation
  • run: execute main test scenario
    • There are sub-phases too??
  • extract: collect data from simulation
  • check: scoreboard tasks. Check for errors based on actual and expected values
  • report: generate reports
  • final: clean up before ending simulation

Transaction level modeling (TLM)

  • Data represented as transactions
  • Not specific to SystemVerilog
  • Transactions are objects that contain random protocol specific info
  • Data flow is via TLM interfaces
  • TLM port: specify an API (set of methods) for a connection
  • TLM export: Actual implementation of a TLM port
  • Analysis ports: Used to monitor data flow
    • Components receiving data from an analysis port doesn't modify that data
    • Can publish data to any number of ports (subscribers ??)
    • https://www.chipverify.com/uvm/uvm-tlm-analysis-port
